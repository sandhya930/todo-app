# Story 1.5: Natural Language Date Parsing

## Status: Draft

## Story

- As any user
- I want to type dates like "next Friday" or "in 3 days" anywhere in my task title
- so that I never have to open a date picker for common scheduling and capture stays fast

## Acceptance Criteria (ACs)

1. The NLP date parser runs automatically on the task title when the user submits a task (text capture, voice review, or API)
2. Recognized date/time phrases are extracted from the title string and set as `due_date` on the task; the phrase is removed from the stored title
3. A parsed date chip appears **below the input field** before save, showing the detected date in plain language (e.g., "Due: Friday, Mar 20"); the user can tap the chip to dismiss it (keeping no date)
4. Supported expressions (minimum at launch): `today`, `tomorrow`, `yesterday` (sets to past — no rejection), `next [weekday]`, `this [weekday]`, `in N days/weeks/months`, `on [month] [day]` (e.g., "on March 20"), `[weekday]` (nearest future), `at [time]` (e.g., "at 3pm" — sets time component), `end of week`, `end of month`
5. Parsing is case-insensitive and tolerates surrounding punctuation (e.g., "call dentist, tomorrow" → extracts "tomorrow")
6. If parsing is ambiguous (e.g., bare "Friday" with no qualifier), the app resolves to the **nearest future** occurrence
7. If no date phrase is detected, `due_date` remains null and no chip is shown — silent, no error
8. The parser works **fully on-device** with no network request
9. Parser is locale-aware for date formats: supports `en-US` and `en-GB` at launch (e.g., "3/20" vs "20/3")
10. The parser is exposed as a shared utility reused by Story 1.2 (voice transcript parsing) and the API (Story 1.4 `due_date` string coercion)

## Dev Notes

### Recommended Library: Chrono.js
- GitHub: https://github.com/wanasit/chrono
- Supports: en-US, en-GB, ja, fr, de, pt, zh (only en-US and en-GB required at launch)
- On-device: yes — pure JavaScript, no server call
- Usage: `chrono.parseDate("call dentist tomorrow at 2pm", referenceDate, { forwardDate: true })`
- `forwardDate: true` handles AC 6 (ambiguous weekday → nearest future)

### Shared Utility Pattern
`parseDateFromTitle` must be a **shared, pure function** (no UI dependencies) importable by:
- Text capture component (Story 1.1 integration)
- Voice review screen (Story 1.2)
- API endpoint handler (Story 1.4)

Suggested file location (update in Change Log if architecture defines otherwise):
`src/utils/nlp/parseDateFromTitle.ts`

### Edge Cases to Handle
| Input | Expected behavior |
|---|---|
| `"buy milk today"` | `dueDate` = today midnight, `cleanTitle` = "buy milk" |
| `"call dentist tomorrow at 2pm"` | `dueDate` = tomorrow 14:00, `cleanTitle` = "call dentist" |
| `"submit report by end of month"` | `dueDate` = last day of current month, `cleanTitle` = "submit report by" → trim trailing "by" (best-effort) |
| `"pick up kids friday"` | `dueDate` = nearest future Friday, `cleanTitle` = "pick up kids" |
| `"in 3 days buy dog food"` | `dueDate` = today + 3 days, `cleanTitle` = "buy dog food" |
| `"buy milk"` | `dueDate` = null, no chip shown |
| `"meeting at 3"` | Ambiguous (3am or 3pm?) — default to 3pm (Chrono.js behavior); show chip for confirmation |
| `"on 3/20"` | en-US: March 20; en-GB: 3rd of the 20th month (invalid) → fall back to no date for en-GB with this format |

### Device Clock Dependency
- Parser uses device clock as reference date — this is correct behavior
- Do not use server time for NLP parsing (would require a network call, violating AC 8)
- Timezone: use device's local timezone for all date calculations

### What This Story Does NOT Include
- Multi-language NLP (French, Spanish, etc.) — English only at launch
- Recurring date phrases ("every Monday", "weekly") — handled in a future recurrence story
- Duration phrases ("for 2 hours") — `estimated_duration_minutes` parsing is a future enhancement
- AI-based date inference (just rule/library-based NLP here)
