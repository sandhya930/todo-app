# Story 1.4: REST API Task Creation

## Status: Draft

## Story

- As Dev (power user / developer)
- I want to create tasks via a documented REST API authenticated with an API key
- so that I can integrate the app with my existing scripts, tools, and automation workflows

## Acceptance Criteria (ACs)

1. `POST /api/v1/tasks` accepts: `title` (required), `due_date` (ISO 8601), `project_id` (UUID), `notes` (string), `tags` (string array), `energy_level` (enum), `estimated_duration_minutes` (integer)
2. Authentication is via `Authorization: Bearer {api_key}` header; requests without a valid key return `401 Unauthorized`
3. Successful creation returns HTTP `201 Created` with the full task object (including server-generated `id` and `created_at`)
4. Invalid requests return HTTP `400 Bad Request` with a structured error body identifying the failing field(s)
5. The API is documented via an OpenAPI 3.0 spec served at `/api/docs` (Swagger UI or Redoc)
6. API keys are generated in Settings → Account → API Keys; user can create, label, and revoke keys
7. Rate limit: 1,000 requests per hour per API key; a `429 Too Many Requests` response includes a `Retry-After` header
8. The created task appears in the user's Inbox and syncs to their devices via the standard CRDT/sync path

## Tasks / Subtasks

- [ ] **Task 1 — API key management** (AC: 2, 6)
  - [ ] 1.1 Design API key schema: `id`, `user_id`, `label` (string, max 50 chars), `key_hash` (bcrypt/SHA-256 of raw key), `created_at`, `last_used_at`, `revoked_at`
  - [ ] 1.2 Key generation: generate cryptographically random 32-byte key; return raw key **once** at creation (never stored in plain text after this point); store hash only
  - [ ] 1.3 Settings UI → Account → API Keys: list existing keys (label + created_at + last_used_at, NOT the raw key), "Create new key" button, "Revoke" action per key
  - [ ] 1.4 On "Create new key": show modal with label input → generate → display raw key with "Copy" button + warning "This key will not be shown again"
  - [ ] 1.5 Write unit tests: key generation produces unique keys, hash stored not raw, revoked key rejected on auth

- [ ] **Task 2 — API authentication middleware** (AC: 2)
  - [ ] 2.1 Implement `authenticateApiKey` middleware: extract Bearer token from `Authorization` header, hash it, look up in `api_keys` table, verify not revoked
  - [ ] 2.2 Attach `user_id` to request context on successful auth
  - [ ] 2.3 Return `401 Unauthorized` with body `{ "error": "invalid_api_key" }` on failure
  - [ ] 2.4 Update `last_used_at` on successful authentication (async, non-blocking)
  - [ ] 2.5 Write unit tests: valid key → auth passes; invalid key → 401; revoked key → 401; missing header → 401

- [ ] **Task 3 — POST /api/v1/tasks endpoint** (AC: 1, 3, 4, 8)
  - [ ] 3.1 Route: `POST /api/v1/tasks` protected by `authenticateApiKey` middleware
  - [ ] 3.2 Request body validation:
    - `title`: required, string, 1–500 chars
    - `due_date`: optional, ISO 8601 date or datetime string
    - `project_id`: optional, valid UUID; must belong to authenticated user if provided
    - `notes`: optional, string, max 10,000 chars
    - `tags`: optional, string array, max 10 items, each max 50 chars
    - `energy_level`: optional, enum `high_focus | low_focus | no_brainer`
    - `estimated_duration_minutes`: optional, positive integer
  - [ ] 3.3 On validation failure: return `400` with body `{ "error": "validation_error", "fields": [{ "field": "title", "message": "Title is required" }] }`
  - [ ] 3.4 On success: call `createTask()` (same service as Story 1.1) with `source: 'api'`; return `201` with full task object
  - [ ] 3.5 Write unit tests: all fields valid → 201 + task object; missing title → 400; invalid energy_level → 400; project_id for another user → 400; notes > 10,000 chars → 400

- [ ] **Task 4 — Rate limiting** (AC: 7)
  - [ ] 4.1 Implement rate limiter keyed on `api_key_id`: 1,000 requests per 60-minute sliding window
  - [ ] 4.2 Use a fast store for rate limit counters (Redis or in-memory with TTL — TBD in architecture)
  - [ ] 4.3 On limit exceeded: return `429 Too Many Requests` with `Retry-After: {seconds_until_reset}` header and body `{ "error": "rate_limit_exceeded", "retry_after": N }`
  - [ ] 4.4 Write unit tests: 1,000 requests → all pass; 1,001st → 429 with Retry-After; window resets after 60 minutes

- [ ] **Task 5 — OpenAPI 3.0 documentation** (AC: 5)
  - [ ] 5.1 Write OpenAPI 3.0 spec (YAML or JSON) for `POST /api/v1/tasks`: path, method, auth scheme (Bearer), request body schema, response schemas (201, 400, 401, 429)
  - [ ] 5.2 Serve Swagger UI or Redoc at `/api/docs`
  - [ ] 5.3 Spec must include example request and response bodies
  - [ ] 5.4 Write test: `/api/docs` returns 200; spec validates against OpenAPI 3.0 schema

- [ ] **Task 6 — Sync integration** (AC: 8)
  - [ ] 6.1 API-created tasks follow the same sync path as manually created tasks — no special handling needed if `createTask()` already enqueues for sync
  - [ ] 6.2 Verify API-created task appears in user's Inbox on their mobile device within sync latency target (≤ 3 seconds per PRD §7.1)
  - [ ] 6.3 Integration test: create task via API → verify appears in simulated second device session

## Dev Notes

### Dependency on Story 1.1
- `createTask()` service must support `source` field (`'manual' | 'voice' | 'email' | 'api'`)
- If Story 1.1 is complete without `source`, add it as a non-breaking field addition (nullable, defaults to `'manual'`)

### API Key Security (from PRD §7.3)
- Raw API keys are shown **once only** at creation — this is critical UX and security requirement
- Keys stored as hashes only (bcrypt or SHA-256 with app-level salt)
- Keys are scoped read-write for this story; read-only scope is a future enhancement (PRD Open Question OQ-7)
- HTTPS is required for all API traffic — document this in the OpenAPI spec server definition

### API Versioning
- All routes under `/api/v1/` — versioning is in the path, not headers, for simplicity
- Breaking changes will require a `/api/v2/` path; this story establishes the pattern

### Rate Limiting (from PRD §6.4)
- 1,000 requests/hour is per API key, not per user (a user with 3 keys gets 3,000 total)
- This story only implements rate limiting for the `POST /api/v1/tasks` endpoint; a global API rate limiter middleware will be added when more endpoints are implemented

### Error Response Format
All API errors must follow this consistent shape:
```json
{
  "error": "error_code_snake_case",
  "message": "Human-readable description",
  "fields": [{ "field": "title", "message": "Title is required" }]
}
```
`fields` is only present for `validation_error`.

### What This Story Does NOT Include
- GET, PATCH, DELETE task endpoints (future story)
- Webhook delivery (Epic 6, Story 6.4)
- Read-only API key scope
- OAuth 2.0 app authorization (for third-party integrations — Phase 3)
- Pagination or listing endpoints

**Manual Test Steps:**
1. Go to Settings → Account → API Keys → "Create new key"
2. Label it "Test Key", click Create → copy the displayed raw key
3. Close modal → verify key appears in list with label (raw key NOT visible)
4. Run: `curl -X POST https://localhost/api/v1/tasks -H "Authorization: Bearer YOUR_KEY" -H "Content-Type: application/json" -d '{"title":"API test task"}'`
5. Verify: 201 response with task object including `id` and `created_at`
6. Open app → verify "API test task" appears in Inbox
7. Revoke the key in Settings → retry the curl → verify 401 response
8. Visit `/api/docs` → verify Swagger/Redoc UI loads and shows the endpoint
