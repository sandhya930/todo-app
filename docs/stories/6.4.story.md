# Story 6.4: Webhooks

## Status: Draft

## Story

- As Dev (power user / developer)
- I want to configure webhooks that fire on task lifecycle events
- so that I can trigger external automations — Slack messages, GitHub issues, custom scripts — whenever my tasks change state

## Acceptance Criteria (ACs)

1. Webhook management is available in Settings → Integrations → Webhooks
2. Supported events: `task.created`, `task.completed`, `task.deferred`, `task.deleted`, `task.updated`
3. Each webhook configuration has: destination URL (required, HTTPS only), a secret (for HMAC-SHA256 payload signing), an event filter (one or more events), and an active/inactive toggle
4. Every delivery includes an `X-Todo-Signature` header containing `sha256=HMAC(secret, raw_body)` so the receiver can verify authenticity
5. Delivery is at-least-once with retry: up to 3 attempts with exponential backoff (1s, 5s, 25s) on non-2xx responses or connection errors
6. A delivery log is available per webhook showing the last 50 deliveries: timestamp, event type, HTTP status code, response time, and a "Resend" button
7. Max 10 webhooks per Pro account; free tier: 0 webhooks (Pro-only feature); upgrade prompt shown on free tier
8. Task `notes` content is NOT included in webhook payloads by default; an explicit "Include notes in payload" opt-in toggle exists per webhook (privacy-first)

## Tasks / Subtasks

- [ ] **Task 1 — Webhook configuration data model** (AC: 1, 3, 7, 8)
  - [ ] 1.1 Create `Webhook` table: `id` (UUID), `user_id` (FK → User), `url` (string, HTTPS only), `secret` (string, stored encrypted at rest), `events` (string array, subset of supported events), `include_notes` (boolean, default false), `active` (boolean, default true), `created_at`, `updated_at`
  - [ ] 1.2 Create `WebhookDelivery` table: `id` (UUID), `webhook_id` (FK → Webhook), `event_type` (string), `payload_hash` (SHA-256 of payload — not the payload itself, for privacy), `http_status` (nullable int), `response_time_ms` (nullable int), `attempt_count` (int), `delivered_at` (nullable timestamp), `last_attempted_at`, `status` (enum: `pending | delivered | failed`)
  - [ ] 1.3 Enforce: URL must be HTTPS (reject `http://`); max 10 webhooks per user (Pro); 0 for free tier
  - [ ] 1.4 Secret stored encrypted at rest (AES-256 or equivalent); never returned in plaintext after initial creation
  - [ ] 1.5 Write unit tests: HTTPS enforcement (http:// rejected), max 10 limit, free tier blocked, secret encryption, `include_notes` defaults false

- [ ] **Task 2 — Webhook management UI** (AC: 1, 3, 6, 7, 8)
  - [ ] 2.1 Settings → Integrations → Webhooks: list of configured webhooks; each shows URL (truncated), active toggle, event chips, "Edit" and "Delete" buttons
  - [ ] 2.2 Free tier: shows "Webhooks are a Pro feature" banner with upgrade CTA; no webhook list
  - [ ] 2.3 "Add webhook" form: URL input, secret input (with "Generate random secret" button), event checkboxes (all 5 events), "Include notes" toggle, active toggle
  - [ ] 2.4 Secret shown once on creation in a copy modal ("Save this secret — it won't be shown again"); after closing, secret is masked in the UI
  - [ ] 2.5 Delivery log: button per webhook opens a panel/drawer showing last 50 `WebhookDelivery` records; each row shows timestamp, event, status chip (green/red), HTTP status, response time; "Resend" button per row
  - [ ] 2.6 Write component tests: list renders all webhooks; free tier banner shown; secret shown once then masked; delivery log table renders; "Resend" fires correct action

- [ ] **Task 3 — Webhook dispatch service** (AC: 2, 4, 5)
  - [ ] 3.1 Create `WebhookDispatchService.dispatch(userId, event, task)`:
    - Find all active `Webhook` records for `userId` where `events` contains `event`
    - For each: build payload (Task 4), sign it (Task 5), enqueue a delivery job
  - [ ] 3.2 Wire dispatch calls into the task service layer:
    - `createTask()` → dispatch `task.created`
    - `completeTask()` → dispatch `task.completed`
    - Deferral job (Story 3.2) → dispatch `task.deferred` per deferred task
    - `deleteTask()` → dispatch `task.deleted`
    - `updateTask()` → dispatch `task.updated` (only if title, due_date, project_id, or energy_level changed — not on every field write)
  - [ ] 3.3 Dispatch is asynchronous and non-blocking: task service operations are not slowed by webhook delivery
  - [ ] 3.4 Write unit tests: dispatch called with correct event on each trigger; non-active webhook not dispatched; webhook with non-matching event filter not dispatched; dispatch is async (does not block task service)

- [ ] **Task 4 — Payload structure** (AC: 2, 8)
  - [ ] 4.1 Standard payload envelope:
    ```json
    {
      "event": "task.completed",
      "timestamp": "2026-03-16T10:00:00Z",
      "user_id": "uuid",
      "task": {
        "id": "uuid",
        "title": "Buy groceries",
        "status": "completed",
        "due_date": "2026-03-16",
        "project_id": "uuid",
        "project_name": "Personal",
        "energy_level": "no_brainer",
        "created_at": "2026-03-10T08:00:00Z",
        "completed_at": "2026-03-16T10:00:00Z"
      }
    }
    ```
  - [ ] 4.2 `notes` field: only included if webhook has `include_notes = true`; otherwise field is absent from payload (not null — absent)
  - [ ] 4.3 `task.deleted` event: task object includes only `id` and `title` (other fields may be unavailable post-deletion)
  - [ ] 4.4 `task.updated` event: payload includes a `changes` object listing only the fields that changed: `{ "changes": { "title": { "from": "old", "to": "new" } } }`
  - [ ] 4.5 Write unit tests: payload structure matches spec; notes absent when `include_notes=false`; notes present when true; changes object in update payload; deleted payload is id+title only

- [ ] **Task 5 — HMAC signing and delivery** (AC: 4, 5)
  - [ ] 5.1 For each delivery: compute `HMAC-SHA256(secret, rawBody)` where `rawBody` is the JSON-serialised payload (UTF-8); set header `X-Todo-Signature: sha256={hex_digest}`
  - [ ] 5.2 Set additional headers: `Content-Type: application/json`, `X-Todo-Event: task.completed`, `X-Todo-Delivery: {deliveryId}`, `User-Agent: TodoApp-Webhook/1.0`
  - [ ] 5.3 Delivery job: `POST {webhook.url}` with signed payload; timeout: 10 seconds
  - [ ] 5.4 On success (2xx): update `WebhookDelivery` → `status=delivered`, record `http_status`, `response_time_ms`, `delivered_at`
  - [ ] 5.5 On failure (non-2xx or timeout): schedule retry with exponential backoff — attempt 2 after 1s, attempt 3 after 5s, attempt 4 after 25s; after 3 failed attempts set `status=failed`; no further retries
  - [ ] 5.6 Write unit tests: HMAC computed correctly (compare against known test vector); all headers set; retry schedule correct (1s, 5s, 25s); status updated on success/failure; no retry after 3rd failure

- [ ] **Task 6 — Delivery log and resend** (AC: 6)
  - [ ] 6.1 `GET /api/v1/webhooks/{id}/deliveries`: returns last 50 `WebhookDelivery` records for a webhook, ordered by `last_attempted_at` desc
  - [ ] 6.2 "Resend" button in delivery log: `POST /api/v1/webhooks/{id}/deliveries/{deliveryId}/resend`; creates a new `WebhookDelivery` record and re-enqueues the delivery job with the original payload (reconstructed from the event type + task_id at time of original event — log the task snapshot or reconstruct from current state)
  - [ ] 6.3 Retain delivery logs for 30 days; older records pruned by a daily cleanup job
  - [ ] 6.4 Write unit tests: delivery log endpoint returns correct records; resend creates new delivery; log pruning removes records older than 30 days

## Dev Notes

### Dependencies
- REST API authentication (Story 1.4) — webhook management endpoints use the same auth
- `createTask()`, `completeTask()`, `updateTask()`, `deleteTask()` services — all need dispatch hooks
- Deferral job (Story 3.2) — extend to dispatch `task.deferred` events
- `EmailSendService` pattern (Story 4.3) — similar async job queue approach for webhook delivery
- Job queue infrastructure (TBD in architecture — BullMQ, pg-boss, or equivalent; same queue used for email sending)

### HMAC Verification Reference (for receiver documentation)
Receivers can verify the signature with:
```python
import hmac, hashlib
expected = 'sha256=' + hmac.new(secret.encode(), raw_body, hashlib.sha256).hexdigest()
assert hmac.compare_digest(expected, request.headers['X-Todo-Signature'])
```
```javascript
const crypto = require('crypto');
const sig = 'sha256=' + crypto.createHmac('sha256', secret).update(rawBody).digest('hex');
const valid = crypto.timingSafeEqual(Buffer.from(sig), Buffer.from(req.headers['x-todo-signature']));
```
Include this in the API docs and webhook setup UI.

### Secret Encryption at Rest (AC 3)
- Webhook secrets are used for HMAC signing; they must be recoverable (not one-way hashed)
- Use AES-256-GCM encryption with a server-side encryption key (stored in environment variables, not DB)
- Decrypt at delivery time only; never store or log in plaintext

### Payload Resend and Task State
For resend (Task 6.2), the original event payload is not stored (only its hash). Options:
1. Store a compressed snapshot of the task at event time in `WebhookDelivery.task_snapshot` (JSONB column, adds storage)
2. Re-fetch current task state for resend (payload will reflect current state, not original)
- **Recommended:** Option 2 for MVP (simpler, less storage); document this limitation in the delivery log UI ("Resend uses current task state")

### `task.updated` Noise Reduction
- Only dispatch `task.updated` for user-visible field changes: `title`, `due_date`, `project_id`, `energy_level`, `status`, `notes`
- Do NOT dispatch for: `today_sort_order`, `pinned_today`, `last_interacted_at`, sync metadata fields
- This prevents webhooks from firing on every CRDT sync write

### Free Tier = 0 Webhooks
- Webhooks are positioned as a Pro power feature
- Free tier users see a prominent upgrade prompt in the Webhooks settings section
- API returns `403 Forbidden` with `{ "error": "pro_required", "feature": "webhooks" }` if a free user calls webhook management endpoints

### What This Story Does NOT Include
- Webhook verification (proving the URL belongs to the user before saving) — future safety feature
- Webhook pausing on repeated failures (auto-disable after N consecutive failures) — future enhancement
- OAuth app authorisation (for third-party apps to receive webhooks on behalf of users) — Phase 3
- Streaming / WebSocket-based events — webhooks only (HTTP push)
- Webhook templates or pre-built integrations (Slack, GitHub) — Phase 3 marketplace
