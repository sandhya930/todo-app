# Story 3.2: Deferral / Forgiveness Mechanic

## Status: Draft

## Story

- As Jordan (ADHD user) and Alex (overwhelmed professional)
- I want tasks I didn't complete to quietly reschedule to the next day rather than showing as "overdue"
- so that opening the app never triggers guilt or avoidance behaviour

## Acceptance Criteria (ACs)

1. The word "overdue" does not appear anywhere in the app UI — in labels, copy, notification text, empty states, or tooltips
2. At midnight (user's local timezone), any task with `due_date = today` and `status != 'completed'` is automatically transitioned to `status = 'deferred'` with `due_date` advanced by 1 day
3. Deferred tasks appear at the top of tomorrow's Today view with a neutral "Carried over" label (soft grey, no red, no exclamation marks)
4. From any view, a deferred task can be: rescheduled to a specific date, moved to Someday, completed, or left as-is (it will appear in Today tomorrow)
5. Completing a deferred task does not penalise the user's streak (EP-04 Story 4.2); the task counts normally toward completion stats
6. If a task is deferred 3 or more times consecutively, the app gently prompts once: "You've carried this over a few times — want to break it into smaller steps or move it to Someday?" (dismissible; prompt shown once per task, not repeatedly)
7. Recurring tasks: a missed occurrence is silently skipped; the next occurrence is created on its normal schedule with no deferred state and no deferral count increment
8. The deferral job is idempotent: running it multiple times for the same midnight window produces the same result (no duplicate deferrals)

## Dev Notes

### Critical: No "Overdue" Anywhere (AC 1)
This is a first-class product principle (PRD §2.2 — guilt/shame as primary retention killer). Before marking story complete:
- Run `grep -ri "overdue\|past due\|missed\|late task\|behind schedule"` across all UI strings, notification copy, email templates, API responses, error messages
- Every instance must be replaced or removed
- Suggested neutral replacements: "deferred" → "Carried over"; "overdue" → (remove entirely or use "Rescheduled"); "missed" → "Moved to tomorrow"

### Timezone Handling (AC 2)
- Job must run at **local midnight per user** — not UTC midnight
- Store `timezone` on User model (IANA timezone string, e.g., `"America/New_York"`)
- If `timezone` is null (not yet set), default to UTC with a fallback warning logged
- DST transitions: when a day is 23 or 25 hours, the job should still run once at local midnight — use a timezone-aware scheduling library

### Idempotency Design (AC 8)
- `last_deferred_at` stores the **calendar date** (not timestamp) of the last deferral in the user's timezone
- Check: `date(last_deferred_at, user.timezone) != today(user.timezone)` before deferring
- This ensures if the job runs twice (retry on failure), the second run is a no-op

### Deferred Count Semantics
- `deferred_count` counts **consecutive deferrals by the midnight job only**
- Manual reschedules do NOT increment `deferred_count`
- Moving to Someday resets `deferred_count = 0` (intentional reset when user makes an active choice)
- Completing resets nothing — historical count preserved for analytics

### Recurring Task Handling (AC 7)
- Recurring tasks are identified by `recurrence_rule IS NOT NULL`
- The deferral job must explicitly skip them — missed occurrences are handled by a separate recurrence engine (future story)
- Document this exclusion in the job's code comments

### What This Story Does NOT Include
- User-configurable deferral time (e.g., defer to next week vs. next day)
- Deferral history / audit trail (future analytics story)
- Recurrence engine for creating next occurrences (separate story)
- Snooze (short-term defer, e.g., "remind me in 2 hours") — future story
