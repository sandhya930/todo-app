# Story 1.3: Email-to-Task Forwarding

## Status: Review

## Story

- As Alex (overwhelmed professional)
- I want to forward an email to a unique address and have it become a task
- so that action items arriving in email are captured in one tap without leaving my email client

## Acceptance Criteria (ACs)

1. Each user account has a unique, static email forwarding address (e.g., `username+todo@app.domain` or a dedicated subdomain)
2. Forwarding an email to that address creates a task: the email **subject line** becomes the task title
3. The email **body** (plain text; HTML stripped to readable Markdown) is stored as the task's `notes` field
4. The task lands in the user's Inbox within 60 seconds of the email being received by the server
5. Only emails sent **from** the user's registered account email address are accepted; all others are silently rejected (no bounce; no task created)
6. The user is notified in-app when an email-created task arrives (if the app is open; silent otherwise)
7. Email subject lines longer than 500 characters are truncated to 500 chars (the task title max)
8. If the email subject is empty, the task title is set to "Email task — [date received]"

## Dev Notes

### Dependency on Story 1.1
- `createTask()` and Task model (including `source` field addition) must be complete
- Add optional `source` field to Task model: `'manual' | 'voice' | 'email' | 'api'` — useful for analytics and filtering. Update 1.1's model if not already present

### Server-Side Service (not client-side)
- This is the first story with a **server-side component** — the email ingestion service runs on the backend
- Architecture docs are not yet defined; the dev agent must select an email receiving provider and document the choice in the Change Log
- Suggested approach: use a managed inbound email webhook service to avoid running an SMTP server

### Security (from PRD §7.3)
- Sender verification (AC 5) is the primary spam/abuse protection — **do not skip this check**
- Silently reject unauthorized emails (no bounce message) to avoid revealing that the address exists
- Rate limit: max 50 email-created tasks per user per day to prevent abuse

### HTML-to-Markdown Conversion
- Use a well-maintained library (e.g., Turndown for JS, html2text for Python)
- Strip: `<script>`, `<style>`, `<img>` (no image storage in scope), inline event handlers
- Preserve: `<a href>` → `[text](url)`, `<ul>/<li>` → `- item`, `<strong>` → `**text**`, `<em>` → `_text_`

### Latency Target (from PRD §7.1 + AC 4)
- Task must appear in Inbox within **60 seconds** of email receipt
- Email webhook → task creation pipeline should target < 10 seconds end-to-end; the 60-second SLA accounts for email delivery delays outside our control

### What This Story Does NOT Include
- Attachments (images, PDFs) — stripped from email; not stored
- CC/BCC to forwarding address — only direct To: is supported at launch
- Unsubscribe / disable forwarding option (Settings only shows the address; disabling is a future story)
- Push notifications for email-created tasks (in-app notification only, as per AC 6)
