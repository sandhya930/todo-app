# Story 5.1: Accountability Partner

## Status: Draft

## Story

- As Alex (overwhelmed professional)
- I want to link a task to an accountability partner who gets notified when I complete or defer it
- so that the social commitment helps me follow through on things I would otherwise procrastinate

## Acceptance Criteria (ACs)

1. An "Add accountability partner" action is available on any task via the task detail view (partner icon button)
2. The partner is invited by email; they receive an opt-in invitation email and do NOT need an app account to receive status notifications
3. The partner sees a read-only view of the linked task (title, due date, status only — NOT notes or sub-tasks)
4. The partner receives an email notification when the task is: completed (celebratory tone) or deferred 2+ consecutive times (supportive tone — never shaming)
5. The task owner can remove the accountability partner at any time; the partner receives a brief "you've been removed" email
6. Free tier: max 5 active accountability-linked tasks per user; Pro: unlimited. Attempting to link a 6th task on free tier shows an upgrade prompt
7. Max 1 accountability partner per task
8. Partner notification emails comply with CAN-SPAM / GDPR: opt-in link, unsubscribe link in every notification, no account required to unsubscribe

## Tasks / Subtasks

- [ ] **Task 1 — AccountabilityLink data model** (AC: 1, 6, 7)
  - [ ] 1.1 Create `AccountabilityLink` table: `id` (UUID), `task_id` (FK → Task, unique — enforces 1 partner per task), `owner_user_id` (FK → User), `partner_email` (string), `partner_name` (optional string), `status` (enum: `pending | active | removed`), `invite_token` (UUID, for opt-in link), `unsubscribe_token` (UUID, for one-click unsub), `opted_in_at` (nullable timestamp), `created_at`
  - [ ] 1.2 Enforce 1 partner per task at DB level (unique constraint on `task_id`)
  - [ ] 1.3 Free tier count check: before creating a link, count `AccountabilityLink` records for `owner_user_id` where `status != 'removed'`; block if ≥ 5 and user is on free tier
  - [ ] 1.4 Write unit tests: unique task_id constraint, free tier count enforcement (5 allowed, 6 blocked), status transitions

- [ ] **Task 2 — Add partner UI in task detail** (AC: 1, 6, 7)
  - [ ] 2.1 Task detail view: "Add accountability partner" button (person+ icon); shown only when no partner is currently linked
  - [ ] 2.2 Tapping opens a sheet: email input (required), optional name input ("So we can personalise the notification"), "Send invite" button
  - [ ] 2.3 Basic email validation before sending; "Send invite" disabled until valid email entered
  - [ ] 2.4 Free tier gate: if user has 5 active links and taps "Add accountability partner" → show upgrade modal before the input sheet
  - [ ] 2.5 After invite sent: button replaced with partner status chip: "⏳ [email] — invite pending" or "✓ [name/email] — active"
  - [ ] 2.6 "Remove partner" action in partner chip kebab menu; confirm dialog: "Remove [name] as your accountability partner for this task?" → sets `status = 'removed'`
  - [ ] 2.7 Write component tests: button shown when no partner; input validation; free tier gate; chip shows pending/active state; remove confirm fires

- [ ] **Task 3 — Invite email and opt-in flow** (AC: 2, 8)
  - [ ] 3.1 On "Send invite": send invitation email to `partner_email` via `EmailSendService`
  - [ ] 3.2 Invitation email content:
    - Subject: "[Owner name] wants you as their accountability partner"
    - Body: "[Owner] is working on: '[task title]' and would like you to cheer them on. You'll get a quick note when they complete it."
    - CTA: "Yes, I'll be their partner" → `GET /api/v1/accountability/accept?token={invite_token}` → sets `status = 'active'`, `opted_in_at = now()` → shows simple confirmation page
    - Footer: "Not interested? Simply ignore this email. [Unsubscribe]" → `unsubscribe_token` link
  - [ ] 3.3 Accepting the invite (no account required — token-only endpoint): returns a branded HTML page "You're [Owner]'s accountability partner for '[task]'. We'll update you when they make progress. ✓"
  - [ ] 3.4 If partner already has an app account (email matches): link `AccountabilityLink` to their `user_id`; they see the task in their own app under "Accountability" section (Phase 2 enhancement — stub as no-op for now)
  - [ ] 3.5 Write unit tests: invite email sent on creation; accept endpoint sets status=active; unsubscribe via invite email token works; already-accepted token is a no-op (idempotent)

- [ ] **Task 4 — Completion notification** (AC: 4)
  - [ ] 4.1 When `completeTask()` is called: check if task has an `AccountabilityLink` with `status = 'active'`; if so, enqueue a completion notification email
  - [ ] 4.2 Completion email:
    - Subject: "🎉 [Owner name] just completed '[task title]'!"
    - Body: "[Owner] did it! They completed '[task]'. Send them a virtual high five."
    - Footer: unsubscribe link (sets `status = 'removed'`, `opted_in_at` cleared)
  - [ ] 4.3 Tone is celebratory; no performance metrics, no "it took N days"
  - [ ] 4.4 Write unit tests: completion email enqueued when link is active; not enqueued when link is pending or removed; unsubscribe from completion email removes link

- [ ] **Task 5 — Deferral notification** (AC: 4)
  - [ ] 5.1 After the midnight deferral job (Story 3.2) runs: for each newly deferred task that has an active `AccountabilityLink`, check if `deferred_count >= 2`
  - [ ] 5.2 If `deferred_count = 2` (exactly — not on every subsequent deferral): enqueue a supportive notification email to the partner
  - [ ] 5.3 Deferral email:
    - Subject: "[Owner name] is still working on '[task title]'"
    - Body: "[Owner] has been carrying '[task]' forward for a couple of days — they're still at it. A word of encouragement goes a long way."
    - Footer: unsubscribe link
  - [ ] 5.4 Tone is explicitly supportive, NOT shaming; copy reviewed by PO before release
  - [ ] 5.5 Do NOT send another deferral notification at deferred_count = 3, 4, etc. — one notification per task, per deferral threshold
  - [ ] 5.6 Write unit tests: notification at deferred_count=2 only (not 1, not 3); not re-sent if already sent (track with `deferral_notification_sent_at` on AccountabilityLink)

- [ ] **Task 6 — Partner removal and removal email** (AC: 5)
  - [ ] 6.1 Owner removes partner: sets `status = 'removed'`; enqueue removal email to partner
  - [ ] 6.2 Removal email: subject "You've been removed as an accountability partner", body "No action needed — you won't receive any more updates for this task.", footer unsubscribe link (for any future invites from any user — scoped to partner email)
  - [ ] 6.3 Unsubscribing from any notification email also sets `status = 'removed'` on the link
  - [ ] 6.4 Write unit tests: removal sets status, removal email enqueued; unsubscribe via notification email removes link

## Dev Notes

### Dependencies
- Task model (EP-01 1.1) — `AccountabilityLink` references `task_id`
- `completeTask()` service (EP-01 1.1) — must check for active link and enqueue completion notification
- Deferral job (EP-03 3.2) — extend to check for active links after deferral and enqueue deferral notifications
- `EmailSendService` (EP-04 4.3) — reuse for all accountability emails
- User model `timezone` (EP-03 3.2) — used for deferral job timing

### No Account Required for Partners (AC 2)
This is a deliberate design choice from the PRD:
- Partners interact only via email links (invite, unsubscribe)
- Token-based endpoints are public (no auth) but single-use or idempotent
- Partner email addresses are stored but never displayed in any public UI
- GDPR: partners are "data subjects" — add a `partner_data_deleted_at` field; if partner requests deletion, null out `partner_email` and set status=removed

### Notification Tone Enforcement (AC 4)
From PRD §2.2 (shame-avoidance principle):
- **Completion email:** celebratory, brief, no performance commentary
- **Deferral email:** warm, supportive, frames the owner positively ("still working on it")
- **Never:** "missed", "failed", "overdue", "still hasn't", "again"
- All notification copy must be reviewed by PO before this story ships

### deferral_notification_sent_at
Add `deferral_notification_sent_at` (nullable timestamp) to `AccountabilityLink`:
- Set when the deferral notification is sent (deferred_count=2)
- Prevents re-sending on deferred_count=3, 4, etc.
- Reset to null if task is completed and re-activated (new accountability cycle)

### Free Tier Enforcement
- Count is on `AccountabilityLink` records with `status IN ('pending', 'active')` — pending invites count toward the limit
- Rationale: prevent circumventing limit by sending invites and never accepting
- Upgrade prompt should cite the value: "Upgrade to Pro to link accountability partners to unlimited tasks"

### What This Story Does NOT Include
- In-app accountability partner experience (partner seeing task in their own app — Phase 2)
- Two-way messaging between owner and partner
- Partner initiating contact or commenting on tasks
- Group accountability (multiple partners per task)
- Partner leaderboards or social graphs
