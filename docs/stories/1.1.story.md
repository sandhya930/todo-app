# Story 1.1: Instant Text Capture

## Status: Review

## Story

- As an overwhelmed professional (Alex)
- I want to capture a task by typing a single line and pressing Enter
- so that I never lose momentum switching to a complex multi-field form

## Acceptance Criteria (ACs)

1. App opens directly to a focused text input field (or a floating action button on the home screen that immediately opens an input)
2. Pressing Enter (keyboard) or tapping "Add" (mobile) saves the task immediately — title is the only required field
3. Task appears in the Inbox view within 100ms of save (local write; no network dependency)
4. Task is persisted to the local database before any network/sync request is initiated
5. Input field clears immediately after save and regains focus, ready for the next capture
6. Maximum title length is 500 characters; a character-count warning appears at 480 characters
7. Task is created with `status: inbox` and `created_at` timestamp; all other fields default to null/unset
8. Task capture works fully offline — queued locally and synced when connectivity is restored

### Core Task Model (from PRD §6.1)

A Task MUST have:
- `id` — UUID v4
- `title` — string, required, max 500 characters
- `status` — enum: `inbox | active | deferred | completed | archived`
- `created_at` — ISO 8601 timestamp (local device time)

A Task MAY have (all nullable/optional for this story):
- `due_date`, `project_id`, `energy_level`, `notes`, `tags[]`, `recurrence_rule`, `estimated_duration_minutes`, `assignee_id`, `accountability_partner_id`

Add implementation field (not in PRD, required for offline sync):
- `pending_sync` — boolean, default `true` on creation; set to `false` after successful server sync

### Offline-First Architecture (from PRD §7.2)

- Local database is the **source of truth** — remote is secondary
- Tasks are persisted locally **before** any network request
- Sync must use CRDT-compatible approach (Yjs / Automerge — TBD); for this story, a simple `pending_sync` flag queue is sufficient as the CRDT layer is not yet implemented
- Target: app fully functional with zero connectivity

### Performance Targets (from PRD §7.1)

- Task save latency (local write): **≤ 100ms**
- App launch to interactive (cold start, mobile): **≤ 2 seconds** — do not block startup on DB initialization

### UI / UX Constraints (from PRD §6.2, §4.1 Persona Alex)

- Today view is the **default home screen** after onboarding
- Quick capture must be **instantly accessible** from the home screen — zero navigation required
- On mobile: FAB + bottom-sheet pattern is standard and familiar
- On desktop/web: persistent input bar above the task list
- Input must clear and refocus immediately — supports rapid multi-task capture ("inbox dump" workflow)

### Security Notes

- `title` input must be sanitized before storage to prevent XSS in web renderer
- No user-generated content should be executed as code at any point

### What This Story Does NOT Include

- Natural language date parsing (Story 1.5 — US-005)
- Voice capture (Story 1.2 — US-002)
- Email forwarding (Story 1.3 — US-003)
- REST API (Story 1.4 — US-004)
- Project assignment during capture (Epic 2)
- Energy level tags during capture (Epic 2)
