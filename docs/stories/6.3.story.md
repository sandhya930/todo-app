# Story 6.3: CLI Companion

## Status: Draft

## Story

- As Dev (power user / developer)
- I want a CLI tool that mirrors the web app's core task management actions
- so that I can create, view, and complete tasks from the terminal without switching context to a browser or mobile app

## Acceptance Criteria (ACs)

1. The CLI is installable via `npm install -g @todo-app/cli` and `brew install todo-app`
2. Authentication: `todo auth login` opens a browser OAuth flow and stores the resulting API token in `~/.todorc`; `todo auth logout` clears the token
3. Core commands are implemented (see command table in Dev Notes)
4. Output supports three formats: human-readable (default, colourised), `--json` (machine-readable), `--csv` (for piping into other tools)
5. The CLI operates offline: reads from a local cache (`~/.todo-cache.db`) when no network is available; writes are queued and synced on next successful command
6. Tab completion is available for zsh and bash; installation instructions printed by `todo completion --help`
7. `todo --help` and `todo <command> --help` print usage documentation; all flags documented
8. The CLI communicates with the app's existing REST API (Story 1.4); no new backend endpoints required beyond what already exists

## Tasks / Subtasks

- [ ] **Task 1 — CLI project scaffold** (AC: 1, 7, 8)
  - [ ] 1.1 Create a new Node.js package `@todo-app/cli` in a `packages/cli/` directory (monorepo) or separate repo; use TypeScript
  - [ ] 1.2 CLI framework: use `commander` or `oclif` (document choice in Change Log); defines command structure, help text, and flag parsing
  - [ ] 1.3 Entry point: `#!/usr/bin/env node` shebang; registered as `todo` binary in `package.json` `bin` field
  - [ ] 1.4 npm publish configuration: public package `@todo-app/cli`; `engines: { node: ">=18" }`
  - [ ] 1.5 Homebrew tap: create a `homebrew-tap` repo with a formula for `todo-app`; formula downloads the npm package via `npm install -g`
  - [ ] 1.6 Write unit tests: CLI entry point loads without error; `--version` outputs correct version; `--help` outputs usage text

- [ ] **Task 2 — Authentication** (AC: 2)
  - [ ] 2.1 `todo auth login`:
    - Generate a random `state` parameter; open `https://app.domain/cli-auth?state={state}` in the default browser
    - Start a local HTTP server on a random port to receive the OAuth callback
    - App redirects to `http://localhost:{port}/callback?token={api_key}&state={state}` after user authenticates
    - Verify `state` matches; store `token` in `~/.todorc` as JSON: `{ "token": "...", "user_email": "..." }`
    - Print "✓ Logged in as [email]"
  - [ ] 2.2 `todo auth logout`: delete `~/.todorc`; print "✓ Logged out"
  - [ ] 2.3 `todo auth status`: print current login status and email if logged in
  - [ ] 2.4 All commands except `auth login` / `auth logout` / `--help` / `--version` check for token; if missing, print "Not logged in. Run `todo auth login`" and exit with code 1
  - [ ] 2.5 Write unit tests: token written to `~/.todorc`; missing token exits with code 1 + helpful message; state mismatch rejected

- [ ] **Task 3 — Core command implementations** (AC: 3, 8)
  - [ ] 3.1 Implement all commands from the table in Dev Notes; each command calls the REST API using the stored token
  - [ ] 3.2 `todo add "Task title"`: `POST /api/v1/tasks` with `{ title, due_date?, project_id?, energy_level? }`; print created task id + title on success
  - [ ] 3.3 `todo list`: `GET /api/v1/tasks?status=today` (today's tasks); display as formatted table: id (short), title, due date, project, energy
  - [ ] 3.4 `todo list --all`: `GET /api/v1/tasks?status=active`; all active tasks
  - [ ] 3.5 `todo done <id>`: `PATCH /api/v1/tasks/{id}` with `{ status: "completed" }`; print "✓ Completed: [title]"
  - [ ] 3.6 `todo defer <id>`: `PATCH /api/v1/tasks/{id}` with `{ status: "deferred" }`; print "→ Deferred: [title]"
  - [ ] 3.7 `todo focus`: `GET /api/v1/tasks/pick-for-me`; display the recommended task full-screen in terminal (clear screen, large ASCII title, "Press Enter to mark complete, Q to quit")
  - [ ] 3.8 `todo projects`: `GET /api/v1/projects`; list all active projects with name, task count, colour indicator
  - [ ] 3.9 Write unit tests: each command sends correct HTTP request; API errors displayed as human-readable messages; `--json` flag returns raw JSON; `--csv` flag returns CSV

- [ ] **Task 4 — Output formatting** (AC: 4)
  - [ ] 4.1 Human-readable (default): use `chalk` for colour; use `cli-table3` or `console-table-lite` for tabular output; truncate long titles at 60 chars with ellipsis
  - [ ] 4.2 `--json`: pipe raw API response JSON to stdout; no colour, no truncation; suitable for `| jq` piping
  - [ ] 4.3 `--csv`: output CSV with headers; suitable for `> tasks.csv` or `| awk`
  - [ ] 4.4 Error output always goes to stderr (not stdout); exit code 1 on error, 0 on success
  - [ ] 4.5 `--no-color` flag disables all chalk styling (also respects `NO_COLOR` environment variable)
  - [ ] 4.6 Write unit tests: human output truncates long titles; JSON output is valid parseable JSON; CSV output has correct headers; errors go to stderr

- [ ] **Task 5 — Offline local cache** (AC: 5)
  - [ ] 5.1 Use SQLite (`better-sqlite3`) as the local cache at `~/.todo-cache.db`; schema mirrors the Task and Project models (read-only subset of fields needed for display)
  - [ ] 5.2 On each successful `GET` command: update local cache with fresh data
  - [ ] 5.3 On network failure: fall back to local cache for read commands (`list`, `projects`); print a warning "⚠ Offline — showing cached data from [timestamp]"
  - [ ] 5.4 Write commands (`add`, `done`, `defer`) offline: queue the operation in a local `pending_operations` table; on next successful network command, flush the queue first
  - [ ] 5.5 Queue flush on startup: if `pending_operations` has items and network is available, apply them before running the user's command
  - [ ] 5.6 Write unit tests: network failure → cache used for reads; write queued when offline; queue flushed on next online command; stale cache shows warning with timestamp

- [ ] **Task 6 — Tab completion** (AC: 6)
  - [ ] 6.1 `todo completion zsh`: outputs a zsh completion script to stdout; user adds `source <(todo completion zsh)` to their `.zshrc`
  - [ ] 6.2 `todo completion bash`: outputs a bash completion script; user adds `source <(todo completion bash)` to their `.bashrc`
  - [ ] 6.3 Completions cover: command names, `--json`/`--csv`/`--no-color` flags, task IDs for `done`/`defer` (fetches from local cache, not live API)
  - [ ] 6.4 `todo completion --help`: prints installation instructions for both shells
  - [ ] 6.5 Write unit tests: completion scripts output valid shell syntax; task IDs resolved from local cache

## Dev Notes

### Full Command Table (AC 3)

| Command | Description | API call |
|---|---|---|
| `todo add "title"` | Create task in Inbox | `POST /api/v1/tasks` |
| `todo add "title" --due tomorrow --project Work --energy high` | Create with options | `POST /api/v1/tasks` |
| `todo list` | Today's tasks | `GET /api/v1/tasks?filter=today` |
| `todo list --all` | All active tasks | `GET /api/v1/tasks?status=active` |
| `todo list --project "Work"` | Tasks in a project | `GET /api/v1/tasks?project_id={id}` |
| `todo done <id>` | Complete a task | `PATCH /api/v1/tasks/{id}` |
| `todo defer <id>` | Defer a task | `PATCH /api/v1/tasks/{id}` |
| `todo focus` | AI pick + terminal focus mode | `GET /api/v1/tasks/pick-for-me` |
| `todo projects` | List all projects | `GET /api/v1/projects` |
| `todo auth login` | Authenticate | Browser OAuth flow |
| `todo auth logout` | Clear credentials | Local only |
| `todo auth status` | Check auth state | Local only |
| `todo completion zsh` | Zsh completion script | Local only |
| `todo completion bash` | Bash completion script | Local only |

### New API Endpoints Required (AC 8 — verify these exist)
Story 1.4 defined `POST /api/v1/tasks`. The CLI also needs:
- `GET /api/v1/tasks` with filter/status query params
- `PATCH /api/v1/tasks/{id}` for complete/defer
- `GET /api/v1/tasks/pick-for-me` (new — may need to be added alongside this story)
- `GET /api/v1/projects`

If any of these don't exist yet, they must be implemented as part of this story. Document in Change Log.

### `~/.todorc` Format
```json
{
  "token": "api_key_here",
  "user_email": "user@example.com",
  "api_base_url": "https://app.domain/api/v1",
  "last_sync": "2026-03-16T10:00:00Z"
}
```
`api_base_url` allows pointing at a self-hosted instance or staging environment.

### `todo focus` Terminal UX
- Clear terminal screen (`process.stdout.write('\x1Bc')`)
- Display task title in large ASCII art or bold large text (use `figlet` or just `chalk.bold` at large terminal width)
- Show: title, due date (if set), project, notes excerpt (first 200 chars)
- Bottom bar: "[Enter] Complete  [D] Defer  [Q] Quit"
- On Enter: call `done <id>`; print "✓ Complete!"; exit
- On D: call `defer <id>`; print "→ Deferred to tomorrow"; exit
- On Q: exit without action

### Security: ~/.todorc Permissions
- Set file permissions to `600` (user read/write only) on creation: `fs.chmodSync(rcPath, 0o600)`
- Never log the token to stdout/stderr
- `--json` output should NOT include the auth token

### Natural Language Dates in CLI (AC 3)
`--due` flag should accept: "tomorrow", "next friday", "2026-03-20", "in 3 days"
Reuse the same `parseDateFromTitle` logic from Story 1.5 if it can be imported as a Node.js module; otherwise re-implement the Chrono.js wrapper in the CLI package.

### What This Story Does NOT Include
- Interactive TUI (terminal UI with menus) — commands only
- Bi-directional real-time sync in CLI (one-shot sync on each command)
- `todo edit` command (opening a task in an editor like `$EDITOR`) — future enhancement
- Windows PowerShell support (bash/zsh only; WSL users can use bash)
- `todo inbox` command (add without date, same as `todo add` with no `--due`)
