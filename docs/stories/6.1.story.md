# Story 6.1: Keyboard Navigation & Command Palette

## Status: Draft

## Story

- As Dev (power user / developer)
- I want full keyboard navigation and a command palette on the web/desktop app
- so that I can manage all my tasks without ever reaching for the mouse

## Acceptance Criteria (ACs)

1. A command palette opens with `⌘K` (Mac) / `Ctrl+K` (Windows/Linux) from any view; it supports fuzzy search across all commands and tasks
2. The command palette supports at minimum: create task, navigate to any view (Today, Inbox, Projects, Someday, Completed), search tasks, and run actions on the currently selected task (complete, defer, edit, move to Someday, delete)
3. Global keyboard shortcuts are active when no text input has focus; all shortcuts are listed in a cheat sheet accessible via `⌘/` or `Ctrl+/`
4. The full default shortcut set is implemented (see shortcut table in Dev Notes)
5. All shortcuts are user-configurable in Settings → Keyboard Shortcuts: each action shows its current binding; clicking a binding opens a capture input for reassignment; reset-to-defaults button available
6. Tab order follows logical reading order in all views (WCAG 2.1 AA); no keyboard focus traps except intentional modal dialogs
7. Keyboard shortcuts do not fire when focus is inside a text input, textarea, or contenteditable element
8. All shortcuts are web/desktop only — mobile apps are excluded from this story

## Tasks / Subtasks

- [ ] **Task 1 — Global keyboard shortcut manager** (AC: 3, 4, 5, 7)
  - [ ] 1.1 Create `KeyboardShortcutManager`: registers global `keydown` listener; maintains a map of `{ action: string → keybinding: string }`; loads bindings from user settings (with built-in defaults as fallback)
  - [ ] 1.2 Before firing any shortcut handler: check `document.activeElement` — if it is an `input`, `textarea`, `select`, or `[contenteditable]`, skip the shortcut (AC 7)
  - [ ] 1.3 Shortcut bindings stored in User settings as a JSON map (`keyboard_shortcuts: { [action]: string }`); merged with defaults at runtime (user overrides win; new defaults auto-appear for new actions)
  - [ ] 1.4 Implement all shortcuts from the default shortcut table (Dev Notes); each shortcut dispatches a named action event consumed by the appropriate view/service
  - [ ] 1.5 Write unit tests: shortcut fires correct action; shortcut blocked when input focused; custom binding overrides default; new default added without losing user customisations

- [ ] **Task 2 — Command palette UI** (AC: 1, 2)
  - [ ] 2.1 Create `CommandPalette` component: full-width modal overlay (centered, ~600px wide on desktop), opens on `⌘K`/`Ctrl+K`, closes on `Escape` or click-outside
  - [ ] 2.2 Search input (auto-focused on open): fuzzy-matches against command list AND task titles simultaneously; shows grouped results: "Commands" section and "Tasks" section
  - [ ] 2.3 Command list (minimum): New Task, Go to Today, Go to Inbox, Go to Someday, Go to [each project], Search, Complete selected task, Defer selected task, Move to Someday, Edit task, Delete task, Open settings, Open keyboard shortcuts
  - [ ] 2.4 Task results: show up to 5 matching tasks by title with their project colour chip and due date; selecting navigates to the task detail
  - [ ] 2.5 Keyboard navigation within palette: `↑`/`↓` moves selection; `Enter` executes selected item; `Escape` closes
  - [ ] 2.6 Context-awareness: "Complete selected task", "Defer selected task" etc. are only shown if a task is currently selected/focused in the underlying view; otherwise hidden
  - [ ] 2.7 Write component tests: opens on ⌘K; closes on Escape; fuzzy search returns correct commands; ↑/↓ navigation works; Enter executes; context commands hidden when no task selected

- [ ] **Task 3 — Default shortcut implementations** (AC: 4)
  - [ ] 3.1 Implement each shortcut from the table (Dev Notes); wire each to its corresponding service call or navigation action:

  | Shortcut | Action | Handler |
  |---|---|---|
  | `N` | New task (open quick capture) | `openQuickCapture()` |
  | `F` | Focus Mode on selected task | `enterFocusMode(selectedTask)` |
  | `E` | Edit selected task | `openTaskDetail(selectedTask, editMode: true)` |
  | `D` | Set due date on selected task | `openDatePicker(selectedTask)` |
  | `P` | "Pick for me" | `triggerPickForMe()` |
  | `⌘/` / `Ctrl+/` | Show shortcut cheat sheet | `openShortcutCheatSheet()` |
  | `J` | Navigate task list down | `selectNextTask()` |
  | `K` | Navigate task list up | `selectPreviousTask()` |
  | `Space` | Complete selected task | `completeTask(selectedTask)` |
  | `Backspace` | Delete selected task (with confirm) | `deleteTask(selectedTask)` |
  | `⌘K` / `Ctrl+K` | Open command palette | `openCommandPalette()` |
  | `G then T` | Go to Today | `navigateTo('today')` |
  | `G then I` | Go to Inbox | `navigateTo('inbox')` |
  | `G then S` | Go to Someday | `navigateTo('someday')` |

  - [ ] 3.2 Chord shortcuts (`G then T` etc.): implement a 500ms chord window — pressing `G` arms the chord; pressing the second key within 500ms fires the action; otherwise `G` is treated as a no-op
  - [ ] 3.3 `J`/`K` navigation: requires a concept of "selected task" in list views — add `selectedTaskId` to view state; visual highlight on selected task (ring/outline)
  - [ ] 3.4 `Space` to complete: show a 300ms undo toast ("Task completed — Undo") before the completion is final; `Ctrl+Z` or tapping "Undo" within that window reverses it
  - [ ] 3.5 Write unit tests: each shortcut fires correct handler; chord window (G+T within 500ms navigates; G alone after 500ms is no-op); J/K updates selectedTaskId; Space shows undo toast

- [ ] **Task 4 — Shortcut cheat sheet** (AC: 3)
  - [ ] 4.1 `⌘/` / `Ctrl+/` opens a modal displaying all shortcuts grouped by category: Navigation, Task Actions, Views, System
  - [ ] 4.2 Each row shows: action label, current keybinding (reflects user customisations), a small "edit" icon that opens the shortcut configurator (Task 5)
  - [ ] 4.3 Cheat sheet is scrollable; closes on `Escape` or click-outside
  - [ ] 4.4 Write component tests: all default shortcuts listed; custom binding shown; edit icon opens configurator

- [ ] **Task 5 — Shortcut configurator in Settings** (AC: 5)
  - [ ] 5.1 Settings → Keyboard Shortcuts: table of all actions with their current bindings; each row has a "Change" button
  - [ ] 5.2 "Change" opens a key-capture input: shows "Press new shortcut…"; captures the next keydown event (single key or modifier+key); rejects: `Tab`, `Escape`, `Enter` (reserved); shows conflict warning if binding already in use
  - [ ] 5.3 On conflict: "This shortcut is already used by [action]. Replace it?" → Yes replaces; No cancels
  - [ ] 5.4 "Reset all to defaults" button with confirmation
  - [ ] 5.5 Changes saved immediately to user settings and sync to other desktop sessions
  - [ ] 5.6 Write component tests: key capture works; conflict detected and warned; replace confirmed; reset-to-defaults restores all bindings

- [ ] **Task 6 — Tab order and focus management** (AC: 6)
  - [ ] 6.1 Audit all views (Today, Inbox, Projects, Task Detail, Settings) for correct tab order: left-to-right, top-to-bottom, matching visual layout
  - [ ] 6.2 Modal dialogs: trap focus within the modal (Tab cycles only within modal elements); `Escape` closes and returns focus to the trigger element

## Dev Notes

### Web/Desktop Only (AC 8)
- `KeyboardShortcutManager` and command palette are **web app only**
- React Native mobile app: no changes from this story
- Desktop web (macOS/Windows/Linux browsers): full implementation

### Shortcut Conflict Prevention
Shortcuts to avoid (reserved by browsers/OS):
- `⌘W` / `Ctrl+W` — closes tab
- `⌘T` / `Ctrl+T` — new tab
- `⌘R` / `Ctrl+R` — refresh
- `⌘L` / `Ctrl+L` — address bar
- `F5`, `F12` — browser refresh/devtools
- `⌘C/X/V` / `Ctrl+C/X/V` — clipboard (reserved when text selected)

Single-letter shortcuts (`N`, `F`, `E`, `D`, `P`, `J`, `K`) only fire when no input is focused — this is enforced by the `activeElement` check in Task 1.

### Selected Task State
`J`/`K` navigation requires a "selected task" concept across list views:
- Add `selectedTaskId: string | null` to view-level state (not global app state — each view manages its own selection)
- On mount: no task selected (null); first `J` press selects the first task
- `selectedTaskId` is cleared when navigating away from the view
- Visual treatment: focused/selected task has a visible highlight ring (not just cursor hover)

### Command Palette Fuzzy Search
- Fuzzy match algorithm: simple substring match is acceptable for v1; upgrade to `fuse.js` or similar for better results if needed
- Commands are static; task results are queried live from local DB
- Debounce task query by 150ms to avoid excessive DB reads while typing

### Undo for Space-Complete (Task 3.4)
- The 300ms undo window is a soft undo — task is marked complete immediately (no delay), but the completion can be reversed
- Undo: calls `uncompleteTask(taskId)` — sets `status` back to previous status, clears `completed_at`
- `uncompleteTask()` must be implemented as part of this story if not already available

### What This Story Does NOT Include
- Vim keybinding mode (full hjkl navigation, command mode) — future enhancement
- Shortcut macros (recording sequences)
- Mobile keyboard shortcuts (physical keyboard connected to iPad — future enhancement)
- Command palette on mobile
