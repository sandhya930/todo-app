# Story 3.4: Goal Decomposition (AI)

## Status: Draft

## Story

- As Alex (overwhelmed professional)
- I want to enter a plain-language goal and have the app break it into actionable milestone tasks
- so that big ambitions become concrete steps I can actually do, without the paralysis of starting from a blank list

## Acceptance Criteria (ACs)

1. "Start from a goal" option is available in two entry points: (a) when creating a new Project (Story 2.3), and (b) as a capture option in the Inbox quick-add area
2. The user enters a plain-language goal string (e.g., "Launch my personal website by end of April")
3. The AI generates 3–7 milestone tasks with: suggested titles, suggested due dates (relative to the goal's deadline if one is detected), and suggested energy levels
4. The user sees a review screen before any tasks are saved: they can approve all, edit individual tasks (title, date, energy), reorder, or delete unwanted suggestions
5. Only tasks explicitly confirmed on the review screen are added to the system; AI suggestions are never auto-saved without user action
6. A privacy notice is shown before the AI call: "Your goal text will be sent to our AI service to generate tasks. [View Privacy Policy]" with an "OK" and a "Cancel" (which aborts the decomposition, no tasks created)
7. If the AI service is unavailable (network error, timeout > 5 seconds): show a graceful error "Goal decomposition isn't available right now — try again later" with an option to manually add tasks instead
8. An explicit "This isn't helpful" feedback button is available on the review screen; tapping it logs a negative feedback event and dismisses without saving

## Tasks / Subtasks

- [ ] **Task 1 — Privacy consent gate** (AC: 6)
  - [ ] 1.1 Before any AI call, show a one-time consent modal: title "AI-Assisted Planning", body text from AC 6, "OK" and "Cancel" buttons, link to Privacy Policy
  - [ ] 1.2 Once user taps "OK": store `ai_goal_decomp_consent = true` on user profile; do not show again
  - [ ] 1.3 If user taps "Cancel": abort flow, return to previous screen; no task created, no AI call made
  - [ ] 1.4 Consent can be revoked in Settings → Privacy → "AI Features" toggle; revoking sets consent to false, re-shows modal on next use
  - [ ] 1.5 Write unit tests: consent modal shown on first use, not shown on subsequent use; cancel aborts; revoke resets consent flag

- [ ] **Task 2 — Goal input UI** (AC: 1, 2)
  - [ ] 2.1 Entry point A — New Project flow (Story 2.3): after entering project name/color/icon, show an optional step: "Want to start with a goal?" with a text input and "Generate tasks" button (skippable via "Skip" or "Add tasks manually")
  - [ ] 2.2 Entry point B — Inbox capture: add "Start from a goal ✨" option below the quick capture input (visible as a secondary action, not the primary CTA)
  - [ ] 2.3 Goal input: multi-line text field, placeholder "e.g. Launch my personal website by end of April", max 500 characters; character counter shown at 400+
  - [ ] 2.4 "Generate tasks" button disabled until goal text is non-empty; shows loading state while AI call is in flight
  - [ ] 2.5 Write component tests: disabled state, loading state, cancel aborts before AI call if consent not yet given

- [ ] **Task 3 — AI decomposition service** (AC: 3, 7)
  - [ ] 3.1 Create `GoalDecompositionService.decompose(goalText: string, projectContext?: { name, deadline }): Promise<SuggestedTask[]>`
  - [ ] 3.2 Call a server-side LLM endpoint (POST `/api/v1/ai/decompose-goal`) — never call the LLM directly from the client; all AI calls go server-side
  - [ ] 3.3 Server-side prompt template:
    ```
    You are a productivity assistant. Break the following goal into 3-7 concrete, actionable milestone tasks.
    For each task provide: title (max 60 chars), suggested_due_date (ISO date relative to goal deadline if present, else null), energy_level (high_focus|low_focus|no_brainer).
    Goal: "{goalText}"
    Project deadline (if any): "{deadline}"
    Return JSON array only. No explanation text.
    ```
  - [ ] 3.4 Parse and validate LLM response: must be a JSON array; each item must have `title` (string), `suggested_due_date` (ISO date or null), `energy_level` (valid enum or null)
  - [ ] 3.5 Timeout: 5 seconds; on timeout or HTTP error → return error state (AC 7)
  - [ ] 3.6 Write unit tests: valid LLM response parsed correctly; malformed JSON handled gracefully; timeout triggers error state; empty array handled (≥1 task required — show error if AI returns empty)

- [ ] **Task 4 — Review screen** (AC: 4, 5, 8)
  - [ ] 4.1 Create `GoalDecompositionReview` screen: header "Review your tasks", subtitle showing the original goal text (truncated to 80 chars)
  - [ ] 4.2 Each suggested task shown as an editable card: title (inline editable), due date (tap to change via date picker), energy level (EnergyLevelPicker component from Story 2.4), delete button (×)
  - [ ] 4.3 Cards are reorderable via drag-and-drop (same component as Today view Story 2.1)
  - [ ] 4.4 "Add N tasks" button at bottom (N = remaining undeleted cards); disabled if all cards deleted
  - [ ] 4.5 "This isn't helpful" link (bottom, subtle): logs `{ event: 'goal_decomp_feedback_negative', goalTextLength: N, suggestionsCount: N }`, dismisses screen, no tasks saved
  - [ ] 4.6 "Add N tasks" tap: calls `createTask()` for each confirmed task, assigns to current project if coming from project entry point, sets `source = 'ai_decomp'`
  - [ ] 4.7 On success: navigate to the project view (if from project) or Inbox (if from Inbox entry point); show toast "N tasks added"
  - [ ] 4.8 Write component tests: editable fields work, delete removes card, reorder updates order, "Add N tasks" count updates dynamically, feedback button fires event without saving

- [ ] **Task 5 — Error and fallback states** (AC: 7)
  - [ ] 5.1 On AI service error or timeout: dismiss loading state, show inline error: "Goal decomposition isn't available right now — try again or add tasks manually"
  - [ ] 5.2 "Try again" button: re-triggers the AI call (same goal text, same consent already given)
  - [ ] 5.3 "Add tasks manually" button: navigates to the project view or Inbox with quick capture pre-focused — no AI, no review screen
  - [ ] 5.4 Write tests: network error → error UI shown; retry fires new API call; manual fallback navigates correctly

## Dev Notes

### Dependencies
- `createTask()` service (EP-01 Story 1.1) — must support `source = 'ai_decomp'`
- Project model (EP-02 Story 2.3) — entry point A integrates into new project flow
- `EnergyLevelPicker` component (EP-02 Story 2.4)
- NLP date parser (EP-01 Story 1.5) — suggested_due_date from AI may be relative; parse if needed
- Consent model: add `ai_goal_decomp_consent` (boolean, default false) to User settings

### Server-Side Only for AI Calls (Security)
- The LLM API key must NEVER be exposed to the client
- All AI requests go through the app's own backend: `POST /api/v1/ai/decompose-goal`
- The backend validates the user's auth token before forwarding to the LLM provider
- Rate limit the AI endpoint: max 10 decompositions per user per day (prevent abuse / runaway costs)

### LLM Provider Selection
- Provider TBD in architecture (OpenAI GPT-4o-mini, Anthropic Claude Haiku, etc.)
- Abstract behind `LLMClient` interface for easy provider swap
- Keep prompt tokens minimal — the server-side prompt template (Task 3.3) is the entire context

### Suggested Task Validation Rules
After parsing LLM response, validate:
- Minimum 1 task, maximum 7 tasks (reject/truncate if outside range)
- Each title: non-empty string, max 60 chars (truncate if longer)
- `suggested_due_date`: valid ISO date or null; must not be in the past (if past, set to null)
- `energy_level`: valid enum value or null (if invalid string, set to null)

### "Source" Field for Analytics
Add `source = 'ai_decomp'` to tasks created from this flow. This distinguishes AI-generated tasks from manual ones in future analytics and helps measure Goal Decomposition feature value.

### Privacy and Data Handling (from PRD §7.3, OQ-7)
- Goal text is sent to the LLM provider server-side — it passes through our servers but is NOT stored after the LLM response is received
- Log only: `goalTextLength` (integer, not the text itself) for analytics
- This must be documented in the Privacy Policy before feature launch

### What This Story Does NOT Include
- Sub-goal nesting (goals within goals)
- AI refinement of existing tasks
- "Improve this task" AI suggestions on individual tasks
- Saving goals as a separate entity (goals are decomposed into tasks only; no Goal model)
- Collaborative goal setting
